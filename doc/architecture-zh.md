# CC-rDeviceAgent 系统整体架构设计

v1.1：新增 FR-10（业务应用控制）相关组件 App Control Service / Health Evaluator / Local Message Broker；扩展 SDK Health/Control/PubSub API；新增 ADR-013~016。

## 一、设计理念与架构原则

### 1.1 设计理念

CC-rDeviceAgent 采用 **"双面三层"** 架构：

- **双面**：北向面对管理后端（设备管理）；南向面对载荷应用（应用基座）
- **三层**：协议接入层、核心服务层、平台抽象层

### 1.2 架构原则

| 原则           | 含义                                                   |
| -------------- | ------------------------------------------------------ |
| **分层解耦**   | 协议、业务、平台三层独立演进，通过 trait/接口契约      |
| **能力可插拔** | 所有可选能力（OTA、容器、TPM）以模块形式注入，按需启用 |
| **状态明确**   | 关键流程显式状态机，无隐式状态                         |
| **可观测优先** | 任何流程必须可被日志、指标、trace 三种方式观察         |
| **安全内建**   | 安全机制在框架层强制，业务模块无需关心加密细节         |
| **降级友好**   | 任何外部依赖（TPM、bootloader、网络）失败都有 fallback |
| **跨平台一致** | 业务逻辑跨平台一致，差异封装在平台抽象层               |
| **资源可控**   | 所有资源（内存、磁盘、CPU、连接数）有上限与配额        |

## 四、各层详细架构

### 4.1 ① 协议接入层（Protocol Layer）

**职责**：处理所有外部通信协议，把网络消息转换为内部命令/事件。

#### 4.1.1 北向接入

| 组件                 | 协议                | 用途                         | 关键能力                     |
| -------------------- | ------------------- | ---------------------------- | ---------------------------- |
| **北向 gRPC Server** | gRPC over mTLS      | 同步控制、文件传输、升级命令 | 流式传输、双向认证、压缩     |
| **MQTT Client**      | MQTT 5.0 over TLS   | 遥测上报、事件推送、配置接收 | QoS、LWT、会话保持、离线队列 |
| **OTLP Exporter**    | OTLP over gRPC/HTTP | 指标/日志/trace 上报后端     | 批量、压缩、重试             |

#### 4.1.2 南向接入

| 组件                 | 协议                       | 用途                                   |
| -------------------- | -------------------------- | -------------------------------------- |
| **南向 gRPC Server** | gRPC over UDS / Named Pipe | 应用注册、数据传输、配置订阅、升级查询 |
| **OTLP Receiver**    | OTLP gRPC（本地）          | 接收应用上报的 metrics/logs/traces     |

#### 4.1.3 内部端点

| 端点           | 用途                                    |
| -------------- | --------------------------------------- |
| `/metrics`     | Prometheus 格式自监控指标（仅本地访问） |
| `/healthz`     | 健康检查                                |
| `/readyz`      | 就绪检查                                |
| `/debug/pprof` | 性能分析（默认关闭）                    |

#### 4.1.4 接入层的关键设计

- **协议无关的内部消息总线**：所有协议层适配后投递到统一的命令/事件总线，下游业务逻辑不感知协议
- **连接管理器**：维护与后端的长连接，自动重连、指数退避
- **会话上下文**：每个请求带有 `RequestContext`（认证身份、租户、trace id）

### 4.2 ② 核心服务层（Core Services）

核心层按业务域划分为三大子域：

#### 4.2.1 核心服务域（Core Services）

| 组件                      | 职责                                                    | 关键依赖                          |
| ------------------------- | ------------------------------------------------------- | --------------------------------- |
| **Control Service**       | 进程启停、命令执行、系统重启关机、状态查询              | Process Manager、Sandbox          |
| **File Transfer Service** | 流式上传/下载、断点续传、完整性校验                     | Storage、Security Center          |
| **Upgrade Engine**        | OTA 状态机、签名验证、A/B 切换、回滚                    | Bootloader、KeyStore、State Store |
| **Config Manager**        | 三层配置（device/agent/app）、版本管理、回滚、订阅通知  | State Store、Security Center      |
| **Telemetry Pipeline**    | 采集器编排、聚合、批量、压缩、离线缓存                  | Sensor Reader、Storage            |
| **App Control Service**   | 运行时应用控制（reload/pause/resume/参数注入/信号下发） |                                   |
| **Heartbeat Service**     | 心跳上报、连接保活、Last Will                           |                                   |

 **域内关键关系**：
 - Control Service 与 Upgrade Engine 在控制平面共享命令通道
 - File Transfer 是 Upgrade Engine 的下层依赖
 - Telemetry Pipeline 与 Config Manager 双向交互（采集器配置由 Config 下发）
 - App Control Service 与 App Lifecycle 协作：生命周期负责"形态变化"（install/start/stop），App Control 负责"运行时调节"（reload/pause/resume/inject）；二者通过共享的 App Registry 视图协同
 - App Control Service 通过本地 Message Broker 向应用推送控制指令（异步流）

#### 4.2.2 应用基座域（Application Platform）

| 组件                 | 职责                                                |
| -------------------- | --------------------------------------------------- |
| **App Registry**     | 应用注册、身份分配、Session Token 管理、能力发现    |
| **App Lifecycle**    | 应用安装、启停、监控、自动重启、资源隔离            |
| **Data Router**      | 应用与后端之间的数据通道路由、Topic 映射、流量整形  |
| **Config Watcher**   | 应用对配置的订阅管理、变更推送                      |
| **Update Notifier**  | 应用升级通知、升级查询代理                          |
| **Health Evaluator** | 收集应用主动上报的健康状态，执行阈值策略，触发联动  |
| **Local Msg Broker** | 进程内 Pub/Sub broker，承载应用间消息与控制指令分发 |

 **域内关键关系**：
 - App Registry 是基座域的"入口"，应用启动必须先注册
 - App Lifecycle 与 Upgrade Engine 协作处理应用升级
 - Data Router 将应用数据复用北向的 gRPC/MQTT 通道（不为每个应用建立新连接）
- Health Evaluator 订阅应用 ReportHealth 上报，连续失败 N 次后通知 App Lifecycle
  执行重启 / 回滚 / 告警（策略由 Config 下发）
- Local Msg Broker 受 Security Center RBAC 控制，主题级 ACL；不暴露任何网络端口
- App Control Service（核心层）通过 Local Msg Broker 将控制指令推送到目标应用

#### 4.2.3 应用控制流（FR-10）

```
  后端                                  Agent 内部                        载荷应用
   │                                       │                                │
   │ gRPC: AppControl(app_id, op, params)  │                                │
   │──────────────────────────────────────▶│                                │
   │                                       │ AuthN/Z + RBAC                 │
   │                                       │ Priority Queue + Dedup         │
   │                                       │ Audit Log                      │
   │                                       │                                │
   │                                       │ Local Msg Broker.publish       │
   │                                       │───────────────────────────────▶│
   │                                       │                                │ 执行
   │                                       │  ReportControlResult           │◀──
   │                                       │◀───────────────────────────────│
   │     CallbackResult (async)            │                                │+   │◀──────────────────────────────────────│                                │
```

#### 4.2.4 横切支撑域（Cross-cutting）

| 组件                  | 职责                                                             |
| --------------------- | ---------------------------------------------------------------- |
| **Security Center**   | 证书管理、签名验证、加解密、密钥轮换、RBAC 决策点                |
| **Audit Chain**       | 审计日志、哈希链、日志上报、防篡改                               |
| **Observability Hub** | 统一的日志、指标、trace 出口；本地导出+远程上报                  |
| **Scheduler / Quota** | 任务调度（升级窗口、采集周期）、资源配额控制、限流               |
| **State Store**       | 统一持久化存储（SQLite + WAL），含任务、配置、应用清单、审计日志 |

**横切层的设计要点**：
- 所有业务组件**通过依赖注入**使用横切组件，不直接 new
- Security Center 是**强制网关**：所有外部消息必须经其验证
- State Store 是**唯一的持久化点**，禁止业务模块直接读写文件

### 4.3 ③ 平台抽象层（Platform Abstraction Layer）

**核心思想**：业务层只依赖 trait/接口，每个平台提供一套实现。

#### 4.3.1 抽象接口清单

| 抽象                  | 描述                                  | 平台实现要点                                          |
| --------------------- | ------------------------------------- | ----------------------------------------------------- |
| **ProcessManager**    | 进程的创建、监控、停止、资源限制      | Linux: cgroup v2；Windows: Job Object；macOS: launchd |
| **NetworkProbe**      | 网络接口枚举、连通性测试、流量统计    | netlink / WMI / route                                 |
| **StorageProvider**   | 文件操作、磁盘信息、配额              | POSIX FS / NTFS API / 嵌入式裸 flash                  |
| **KeyStore**          | 私钥/凭证安全存储                     | TPM 2.0 / DPAPI / Keychain / 加密文件                 |
| **BootloaderAdapter** | 槽位查询、设置 trial、mark good、回滚 | RAUC / U-Boot env / GRUB / Windows BCD                |
| **SensorReader**      | CPU/内存/温度/电源等指标读取          | sysinfo + 平台专用 API                                |
| **PowerManager**      | 电源状态、关机、重启、休眠            | systemd / Win32 API / IOKit                           |
| **SandboxRunner**     | 命令隔离执行                          | seccomp+namespace / AppContainer / 无（嵌入式）       |
| **ServiceManager**    | 系统服务管理（agent 自身和被管服务）  | systemd / SCM / launchd                               |
| **TimeSource**        | 单调时钟、墙钟、NTP 状态              | 各平台时间 API                                        |

## 五、关键架构模式

### 5.1 命令-事件总线模式（内部）

```
                ┌──────────────────────────┐
                │   Command/Event Bus      │
                │   (in-process async)     │
                └───────────┬──────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   ┌─────────┐         ┌─────────┐         ┌─────────┐
   │ 协议层   │         │ 核心层   │         │ 横切层   │
   │ 投递命令 │         │ 处理&发  │         │ 订阅审计 │
   │         │         │ 出事件   │         │ 与监控   │
   └─────────┘         └─────────┘         └─────────┘
```

**收益**：
- 协议层与业务层解耦
- 审计、监控、调试可以无侵入地订阅所有命令/事件
- 便于做 replay 和故障注入测试

### 5.2 状态机模式（关键流程）

所有"长流程"使用显式状态机：

| 流程         | 状态机                                                   |
| ------------ | -------------------------------------------------------- |
| OTA 升级     | 14 个状态（见前文设计）                                  |
| 应用生命周期 | Registered → Installed → Running → Stopped → Uninstalled |
| 连接管理     | Disconnected → Connecting → Connected → Reconnecting     |
| 配置变更     | Pending → Validating → Applying → Applied / Failed       |
| 文件传输任务 | Queued → Transferring → Verifying → Completed / Failed   |

**统一状态机框架**特性：
- 状态持久化到 State Store
- 状态变更产生事件
- 崩溃后从持久化状态恢复
- 状态转换可视化（生成 mermaid 图）

### 5.3 流水线模式（数据处理）

遥测、日志、应用数据均采用流水线：

```
Source → Buffer → Processor → Buffer → Exporter
                      │
                      ├── Filter（过滤）
                      ├── Aggregator（聚合）
                      ├── Enricher（打标）
                      └── Compressor（压缩）
```

**收益**：
- 各阶段可独立扩展和测试
- 背压（backpressure）可控
- 配置驱动，可动态调整

### 5.4 责任链模式（请求处理）

入站请求经过统一的责任链：

```
Request 
  → AuthN (mTLS 证书提取)
  → AuthZ (RBAC 决策)
  → RateLimit (限流)
  → AuditLog (审计入口)
  → Validate (参数校验)
  → Handler (业务处理)
  → AuditLog (审计出口)
Response
```

### 5.5 Control Service 内部责任链

```
 入站
   → AuthN（mTLS 身份）
   → AuthZ（RBAC 决策点）
   → 命令白名单校验（命令模板 + 参数 schema）
   → Priority Queue（FR-10.7：高优先级抢占；可取消）
   → Dedup / Cancel（基于 command_id 幂等与撤销）
   → Executor（同步执行 / 派发到 Local Msg Broker 异步交付）
   → Audit Chain（审计含操作者、目标、参数摘要、结果、回调状态）
   → Async Callback（FR-10.9：执行结果异步回执后端）
```

### 5.5 适配器模式（平台抽象）

平台抽象层全部采用适配器模式，业务依赖抽象 trait，运行时根据平台注入实现。

### 5.6 仓储模式（状态存储）

业务层使用 Repository 接口操作持久化数据，不直接接触 SQLite/sled。便于：
- 替换存储引擎
- Mock 测试
- 添加缓存层

## 十、可靠性架构

### 10.1 可靠性手段

| 层级       | 手段                                                        |
| ---------- | ----------------------------------------------------------- |
| **进程级** | systemd/SCM 监控 + watchdog；崩溃自动重启；最大重启频率限制 |
| **任务级** | 关键任务 panic 隔离；任务级别看门狗                         |
| **状态级** | 所有关键流程显式状态机 + 持久化 + 崩溃恢复                  |
| **数据级** | SQLite WAL + fsync；写入双拷贝；CRC 校验                    |
| **通信级** | 自动重连 + 指数退避；离线队列；QoS 保证                     |
| **升级级** | A/B 槽位 + trial 模式 + 自动回滚；防变砖                    |
| **资源级** | 内存/磁盘/CPU 配额；超限触发降级而非崩溃                    |

### 10.2 降级策略

| 故障                  | 降级                                         |
| --------------------- | -------------------------------------------- |
| 后端不可达            | 本地缓存遥测、缓存控制命令、保持基本管理能力 |
| TPM 不可用            | 降级为加密文件 + 设备指纹绑定                |
| Bootloader 不支持 A/B | 降级为应用级升级 + agent 自升级              |
| 磁盘空间不足          | 暂停遥测采集、丢弃旧日志、拒绝新升级         |
| 内存压力              | 缩小缓冲区、降低采集频率                     |
| CPU 过载              | 延长采集间隔、暂停非关键采集器               |

### 10.3 自愈机制

 | 检测                          | 自愈动作                                        |
 | ----------------------------- | ----------------------------------------------- |
 | 进程未响应                    | systemd watchdog 触发重启                       |
 | 状态机卡在某状态超时          | 进入失败分支或回滚                              |
 | 持久化队列满                  | 按策略丢弃最旧数据 + 上报告警                   |
 | 证书即将过期                  | 主动触发轮换                                    |
 | 升级后健康检查失败            | 自动回滚到上一版本                              |
 | 配置变更后业务异常            | 自动回滚到上一版本                              |
 | 业务应用连续 N 次健康上报失败 | 按策略：重启 → 回滚 → 告警（升级序列，FR-10.5） |
 | 应用 Pause 超过最大保持时长   | 强制 Resume 或终止，并审计                      |
 | 控制指令队列堆积超阈值        | 拒绝新低优指令 + 告警                           |

## 十二、扩展性架构

### 12.1 扩展点

| 扩展点                | 形式           | 典型扩展                                   |
| --------------------- | -------------- | ------------------------------------------ |
| **采集器**            | trait + 注册表 | 自定义指标采集（GPU、特殊传感器）          |
| **遥测处理器**        | trait + 流水线 | 数据脱敏、自定义聚合                       |
| **遥测导出器**        | trait          | Kafka、HTTP webhook、本地文件              |
| **升级策略**          | trait          | OCI 容器、WASM、自定义包格式               |
| **Bootloader 适配器** | trait          | 自研 bootloader                            |
| **认证后端**          | trait          | LDAP、OAuth、自定义                        |
| **配置存储后端**      | trait          | etcd、Consul、自定义                       |
| **平台抽象**          | trait          | 新平台支持                                 |
| **Health Evaluator**  | trait          | 自定义健康判定（滑动窗口、加权、复合条件） |
| **Control Handler**   | trait          | 自定义控制指令处理器（按 op 路由）         |

### 12.2 扩展模式

- **编译期扩展**：通过 Cargo feature flags，编译时启用，性能最优
- **运行期扩展**：动态库 / WASM 插件，灵活但安全风险高
- **进程外扩展**：通过本地 gRPC，最安全但有 IPC 开销

**推荐策略**：
- 核心扩展点（采集器、升级策略）：编译期 + 运行期可选
- 业务扩展：通过应用基座（载荷应用）实现，不进 agent 进程
- 安全敏感扩展：禁止运行期加载

### 12.3 协议演进

- **Protobuf 版本化**：包名带 v1/v2
- **能力协商**：连接建立时交换支持的协议版本和特性集
- **向后兼容**：N+2 版本兼容策略（支持当前及前两个版本）
- **灰度切换**：新协议通过 feature flag 启用

```
                │  - 应用密钥              │
                │  - 应用特征开关          │
                └─────────────────────────┘
```

### 13.2 配置来源优先级

```
默认值（编译内置）
  ↓ 覆盖
打包配置（/opt/cc-ragent/etc/config.toml）
  ↓ 覆盖
用户配置（/etc/cc-ragent/overrides.d/*.toml）
  ↓ 覆盖
远程配置（从后端下发，持久化于 state.db）
  ↓ 覆盖
环境变量（CC_RAGENT_*）
  ↓ 覆盖
命令行参数（仅启动时）
```

### 13.3 配置生效模式

| 模式             | 含义               | 适用配置                 |
| ---------------- | ------------------ | ------------------------ |
| **热加载**       | 立即生效，无重启   | 日志级别、采集间隔、限速 |
| **重新连接**     | 重建外部连接后生效 | 后端地址、证书           |
| **重启生效**     | 需要 agent 重启    | 监听端口、运行用户       |
| **下次升级生效** | 配合升级流程生效   | 升级策略、签名公钥       |

## 十四、关键架构决策（ADR 摘要）

| 编号    | 决策                                                | 理由                                                |
| ------- | --------------------------------------------------- | --------------------------------------------------- |
| ADR-001 | 选用 Rust 实现                                      | 内存安全、零开销抽象、跨平台、生态成熟              |
| ADR-002 | 单进程多任务模型                                    | 资源占用低，适合嵌入式；通过 sandbox 处理隔离需求   |
| ADR-003 | gRPC + MQTT 双通道                                  | gRPC 适合同步控制，MQTT 适合海量遥测，互补          |
| ADR-004 | SQLite 作为唯一持久化                               | 事务、崩溃安全、零依赖                              |
| ADR-005 | OpenTelemetry 作为可观测性标准                      | 生态最广，避免重复造轮子                            |
| ADR-006 | 平台抽象层使用 trait                                | 编译期多态，运行时零开销                            |
| ADR-007 | OTA 显式状态机 + 持久化                             | 升级是最复杂的流程，可靠性要求最高                  |
| ADR-008 | A/B 槽位作为系统升级首选方案                        | 业界成熟方案，防变砖                                |
| ADR-009 | 应用通过本地 IPC 而非动态库                         | 故障隔离、跨语言友好                                |
| ADR-010 | Ed25519 作为签名算法                                | 性能好、密钥小、无随机数依赖                        |
| ADR-011 | mTLS 作为传输安全基础                               | 双向认证、密钥协商、广泛支持                        |
| ADR-012 | 不内置规则引擎、不内置 ML 推理                      | 这些是应用层职责，保持 agent 轻量                   |
| ADR-013 | 本地应用间通信采用进程内 Broker（非独立 MQTT/NATS） | 降低外部依赖、复用 agent 鉴权与审计、不暴露网络端口 |
| ADR-014 | App Control 与 App Lifecycle 分离为两个服务         | 状态机（重）与运行时控制（轻）职责单一，避免耦合    |
| ADR-015 | 健康策略仅支持阈值/计数器型，不演进为通用规则引擎   | 守住 agent 轻量边界，复杂联动由后端或应用承担       |
| ADR-016 | 控制指令使用优先级队列 + 去重，不使用直通执行       | 防抖动、支持抢占与取消、便于审计与限流              |

## 十五、架构边界（明确不做什么）

**Agent 不负责**：

- ❌ 业务逻辑（属于载荷应用）
- ❌ 规则引擎、复杂事件处理（CEP）
- ❌ AI/ML 推理（属于载荷应用）
- ❌ 设备协议解析（Modbus、CAN、BACnet 等，作为应用插件）
- ❌ 长期数据存储（仅短期缓存）
- ❌ 用户管理（依赖后端）
- ❌ CA 证书签发（依赖外部 PKI）
- ❌ 容器编排（仅启停，编排是后端职责）
- ❌ UI（仅暴露 API）
- ❌ 跨设备的应用间通信（本地 Broker 仅服务于同设备同 agent 下应用；跨设备走后端）
- ❌ 复杂控制编排（DAG / 工作流；agent 仅支持单步指令 + 简单批量）
**架构边界的意义**：保持 agent 轻量、聚焦、可维护。
**关于"业务健康联动"的边界澄清**（FR-10.5）：
- Agent 支持的健康策略形式：**阈值型 / 计数器型 / 滑动窗口型**
- 不支持的形式：跨应用条件组合、时间序列模式匹配、规则 DSL
- 复杂联动语义请由后端或专门的应用承担

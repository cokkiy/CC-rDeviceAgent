# 平台抽象层（Platform Abstraction Layer, PAL）详细设计

---

## 一、设计目标与原则

| 目标             | 说明                                                            |
| ---------------- | --------------------------------------------------------------- |
| **业务无感**     | 上层业务代码不出现 `#[cfg(target_os)]`，平台差异由 PAL 完全吸收 |
| **零运行时开销** | 编译期多态（trait + 泛型），不引入动态分发开销                  |
| **能力可探测**   | 启动时探测平台能力，业务按 CapabilityProfile 决策               |
| **降级友好**     | 高级能力不可用时，自动 fallback 到通用实现                      |
| **测试可替换**   | 提供 Mock 实现，单元测试不依赖真实系统                          |
| **新平台易扩展** | 新增平台只需实现 trait 集合，不动业务代码                       |

## 十、关键设计决策

| 决策             | 选择                            | 理由                                   |
| ---------------- | ------------------------------- | -------------------------------------- |
| **多态方式**     | trait object (`Arc<dyn Trait>`) | 灵活注入，运行期可替换；性能损失可忽略 |
| **错误统一**     | PAL 自定义错误层 + 透传原始错误 | 业务层错误处理统一，调试时仍可溯源     |
| **异步策略**     | 全部 `async` trait              | 与 Tokio 运行时一致，避免阻塞          |
| **能力探测时机** | 启动时探测，结果缓存            | 平衡启动速度与准确性                   |
| **降级开关**     | 配置可控 + 降级事件上报         | 让运维知晓系统运行在何种降级态         |
| **Feature 粒度** | 按平台 + 按能力双维度           | 既能裁剪平台，也能裁剪能力             |
| **测试策略**     | Mock 适配器 + 集成测试矩阵      | 单元测试不依赖真实系统                 |

## 十二、质量属性保障

| 属性         | PAL 设计保障                                         |
| ------------ | ---------------------------------------------------- |
| **可移植性** | 契约统一 + 适配器隔离 + 降级兜底                     |
| **可测试性** | Mock 适配器 + trait 注入                             |
| **性能**     | trait 对象动态分发开销 < 1ns，可忽略；热路径可用泛型 |
| **可维护性** | Crate 边界清晰，新增平台只动一个 crate               |
| **安全性**   | 平台敏感 API 收口在 PAL，业务层无法绕过              |
| **可观测性** | PAL 内部统一打 trace，便于定位平台相关问题           |
| **演进性**   | 契约与实现分离，可独立迭代                           |

---

## 总结

平台抽象层 (PAL) 是整个 CC-rDeviceAgent 跨平台能力的**架构基石**，通过 **契约层 / 能力路由 / 适配器层 / 降级层** 四层结构，实现：

- ✅ **业务层完全平台无关**（不出现 `#[cfg]`）
- ✅ **能力可探测、可降级**（CapabilityProfile 驱动）
- ✅ **新平台易扩展**（实现 trait 集合即可）
- ✅ **测试友好**（Mock 适配器）
- ✅ **安全收口**（平台 API 集中管控）

建议作为 v0.5 阶段的**首要交付物**，为后续 Upgrade Engine、Security Center 等核心服务的跨平台落地提供地基。

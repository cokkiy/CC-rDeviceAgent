## 2. 产品定位

CC-rDeviceAgent 是部署在 **IoT 设备、边缘节点、远程工作站**上的**轻量级、安全的、跨平台的基础代理软件**。提供两大类能力：

- **L1 设备代理能力**：被管理的设备的"管理面"——遥测、控制、升级、配置
- **L2 应用基座能力**：为运行在设备上的业务应用提供"数据面"——指标上报、数据传输、配置分发、热更新

## 3. 范围与术语

### 3.1 范围
- **包含**：Agent 进程本体、本地 SDK（应用基座接口）、Protobuf/OTLP 协议定义
- **不包含**：后端管理平台、应用商店、CA 颁发系统（依赖外部）

### 3.2 术语
- **Agent**：本软件本身
- **后端**：与 Agent 通信的云端/中心管理平台
- **载荷应用 (Payload App)**：在设备上运行、使用 Agent 基座能力的业务应用
- **OTA**：Over-The-Air，远程升级
- **租户**：逻辑隔离单元，一台设备可承载多个租户的应用

## 5. 非功能需求

### 5.1 性能（按设备分级，详见第四章 4.1）

### 5.2 可移植性
- **NFR-PORT-1**：核心代码无平台条件编译，平台差异封装在 `platform` 模块的 trait impl 中
- **NFR-PORT-2**：CI 必须覆盖 linux-x86_64、linux-aarch64、linux-armv7、windows-x86_64、macos-aarch64

### 5.3 可维护性
- **NFR-MAINT-1**：核心模块单元测试覆盖率 ≥ 70%
- **NFR-MAINT-2**：关键流程（升级、配置、传输）集成测试覆盖率 ≥ 80%
- **NFR-MAINT-3**：公开 API 必须有 rustdoc 注释和示例

### 5.4 兼容性
- **NFR-COMPAT-1**：gRPC/Protobuf 接口遵循语义化版本，向后兼容至少 2 个大版本
- **NFR-COMPAT-2**：MQTT topic 结构变更必须可通过配置回退到旧格式

### 5.5 部署
- **NFR-DEPLOY-1**：单二进制部署，无外部依赖（静态链接）
- **NFR-DEPLOY-2**：提供 systemd unit、Windows Service、Docker 镜像、deb/rpm 包
- **NFR-DEPLOY-3**：零停机重启（socket activation / handoff）

## 6. 接口规范

### 6.1 北向接口（与后端）
- gRPC services：`DeviceControl`, `FileTransfer`, `Upgrade`, `Configuration`, `Heartbeat`
- MQTT topics：`{tenant}/{device_id}/telemetry/{category}`, `{tenant}/{device_id}/events`, `{tenant}/{device_id}/lwt`

### 6.2 南向接口（与应用）
- 本地 gRPC over UDS：`AppRegistry`, `Metrics`(OTLP), `Logs`(OTLP), `Config`, `Data`, `Update`
- OTLP HTTP 兼容端点（可选）

### 6.3 平台接口
- Trait `Platform`：进程管理、网络、存储、传感器、电源、密钥存储

## 7. 风险与依赖

| 风险                       | 影响 | 缓解                                        |
| -------------------------- | ---- | ------------------------------------------- |
| Rust 嵌入式工具链不完善    | 中   | 主推 Linux + glibc/musl，特殊嵌入式按需评估 |
| TPM 在不同硬件上接口不一致 | 中   | 抽象 KeyStore trait，提供 fallback 实现     |
| OTA 升级中断导致设备变砖   | 高   | A/B 分区 + 事务化 + 强制健康检查            |
| 后端协议变更               | 中   | gRPC 版本化 + feature negotiation           |
| 多语言 SDK 维护成本        | 中   | 先 Rust + Python，其他按社区贡献            |

## 8. 验收标准

- 所有 P0 需求实现并通过测试
- 安全渗透测试（mTLS、命令注入、路径穿越、重放攻击）通过
- 性能基准达成第 4.1 节指标
- 在至少 3 类目标硬件（树莓派、x86 边缘盒、Windows 工作站）上完成 7×24 小时稳定性测试
- 100 次 OTA 升级测试（含模拟断电）零失败
